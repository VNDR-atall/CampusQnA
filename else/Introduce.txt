## 一、选题定义：校园百事通（公开信息智能问答）

| 项目       | 说明                                                         |
| -------- | ---------------------------------------------------------- |
| **产品名称** | 暂定「XX 大学百事通」或「校问」                                          |
| **核心功能** | 学生用自然语言提问校园办事、生活、学习相关公开信息，系统通过 RAG 检索官方公开文档并生成准确回答（带来源引用）。 |
| **用户门槛** | **无需登录/注册**，打开小程序直接问。                                      |
| **数据来源** | 学校官网公开页面、部门网站通知、学生手册 PDF、后勤公告等**完全公开可爬取**的信息。              |
| **合规性**  | 所有数据均为公开信息，不涉及个人隐私，符合《个人信息保护法》及学校网络管理规定。                   |
| **上线策略** | 小程序前端 + 个人网站 API，无需与校方做技术对接，仅需告知（或备案）即可。                   |

**创新点**：
1. 将分散在几十个学校二级网站的信息，通过 RAG 统一为**一问一答**。
2. 回答**带出处链接**，点击可跳转原文验证，增强可信度。
3. 轻量级，无需用户系统，非常适合作为软件工程实验的**首个可上线作品**。

### 技术栈选型详表（针对百事通场景优化）

| 组件 | 选型 | 在本项目中的作用 | 并发支撑说明 |
|------|------|------------------|--------------|
| **后端框架** | FastAPI | 提供 `/api/chat` 问答接口、管理后台 API | 异步非阻塞，单 Worker 可承载 500+ 并发问答请求 |
| **向量数据库** | ChromaDB（嵌入式模式） | 存储文档切片向量及原文元数据 | 数据量在 5000 条切片以内时，毫秒级检索；无需独立服务 |
| **嵌入模型** | `BAAI/bge-small-zh-v1.5` | 将用户问题、文档切片转为 512 维向量 | CPU 推理，一次编码约 20ms，可本地部署，完全免费 |
| **大模型 API** | DeepSeek-V3 或 通义千问-Turbo | 根据检索到的文档上下文生成自然语言回答 | 免费额度充足（DeepSeek 新用户 500 万 tokens），支持高并发异步调用 |
| **关系数据库** | PostgreSQL（或 SQLite） | 存储问答日志（用于分析热点问题）、用户反馈 | 初期 SQLite 完全够用，后续无缝迁移 PostgreSQL |
| **缓存** | Redis | 缓存高频问题的答案（问题哈希 → 回答），降低 API 调用和检索开销 | 命中率高时，可将 QPS 提升 10 倍以上 |
| **任务队列** | 后台线程 / Celery（可选） | 异步记录问答日志，避免阻塞主流程 | 前期用 FastAPI 的 `BackgroundTasks` 即可 |
| **爬虫** | `requests` + `BeautifulSoup` + `pdfplumber` | 定期抓取官网更新内容，更新知识库 | 单独脚本，每周手动或 Cron 执行 |
| **容器化** | Docker Compose | 一键编排 FastAPI、Redis、ChromaDB、PostgreSQL | 便于部署到你现有服务器 |

## 四、数据准备：知识库构建流程（关键！）

既然数据来自公开渠道，我们需要明确**爬什么、怎么切、如何维护**。

### 1. 可爬取的数据源清单（示例）

| 类别 | 数据源 URL 示例 | 内容类型 | 预计文档数 |
|------|----------------|----------|------------|
| 学生办事指南 | 学生处官网 / 服务大厅栏目 | HTML | 30+ 页面 |
| 后勤服务 | 后勤集团网站（报修、食堂、宿舍） | HTML/PDF | 20+ 条目 |
| 教务常见问题 | 教务处网站“常见问题”板块 | HTML | 50+ 问答对 |
| 图书馆服务 | 图书馆网站（开馆时间、借阅规则） | HTML | 10+ 页面 |
| 校园地图/设施 | 学校概况页面（位置、电话） | HTML | 15+ 条目 |
| 学生手册 | 学生手册 PDF（校规、奖惩） | PDF（文字版） | 1 份，约 100 页 |
| 校历 | 教务处发布的学年校历 PDF/图片 | 需转文字 | 1 份 |

**总计初始文档切片预估**：约 **200~500 条**，完全足够 RAG 演示。

### 2. 文档处理与向量化流水线

```
[官网 HTML] ──► 提取正文 (BeautifulSoup) ──► 清洗格式 ──► 按标题/段落切分
[PDF 文件] ──► 提取文字 (pdfplumber) ──► 按页/章节切分
                                                      │
                                                      ▼
                                          [文本切片] (每片 300~500 字)
                                                      │
                                                      ▼
                                      [嵌入模型 bge-small-zh] 生成向量
                                                      │
                                                      ▼
                                          [存入 ChromaDB]
                                          (附带元数据: 来源URL、标题、更新时间)
```

### 3. 检索增强生成（RAG）核心流程

```
用户问题: "宿舍空调坏了怎么报修？"
        │
        ▼
[向量化问题] ──► 在 ChromaDB 中检索 Top 3 相关切片
        │
        ▼
[构建 Prompt]
"""
你是一个校园助手，请根据以下参考资料回答用户问题。
如果资料中没有明确答案，请如实告知，不要编造。
参考资料：
[切片1] 来源：后勤报修流程页面。内容：...
[切片2] 来源：宿舍管理规定。内容：...

用户问题：宿舍空调坏了怎么报修？
"""
        │
        ▼
[调用 DeepSeek API] ──► 生成回答 + 附带引用链接
        │
        ▼
返回给小程序: { "answer": "...", "sources": ["链接1", "链接2"] }
```

## 五、应对校园级并发的具体策略

### 1. 请求链路优化

| 阶段         | 优化手段                                   | 预期效果                |
| ---------- | -------------------------------------- | ------------------- |
| **问题向量化**  | 嵌入模型本地部署，CPU 多线程推理                     | 单次 < 30ms，无需 API 费用 |
| **向量检索**   | ChromaDB 使用 HNSW 索引                    | Top 3 检索 < 10ms     |
| **大模型生成**  | 使用 DeepSeek 等支持**流式输出**的 API，边生成边返回    | 首字延迟 < 500ms，用户体验流畅 |
| **高频问题缓存** | 将问题文本哈希后存入 Redis，回答缓存 1 小时             | 命中缓存时直接返回，跳过所有计算    |
| **限流保护**   | Nginx `limit_req_zone` 限制单 IP 每秒 5 次请求 | 防止恶意刷接口             |
|            |                                        |                     |

### 2. 容量估算（日活估摸 2000 人）

| 指标 | 计算 | 结论 |
|------|------|------|
| 日均问答量 | 2000 人 × 人均 5 问 = 10000 次 | 平均 QPS ≈ 0.12，峰值 QPS 按 10 倍计 ≈ 1.2 |
| 单次问答耗时（无缓存） | 向量化 30ms + 检索 10ms + LLM 生成 1500ms ≈ 1.5s | 若使用流式输出，用户感知延迟更低 |
| 所需 Worker 数 | QPS 1.2 × 平均耗时 1.5s ≈ 2 并发连接 | **单机单 Worker 即可轻松应对** |

> **结论**：对于校园级应用，初期根本无需考虑复杂集群，一台 2 核 4G 的轻量云服务器即可稳定运行。你的个人网站服务器只要配置不极低，完全够用。

## 六、分阶段开发计划（单人 8 周）

| 周次 | 阶段 | 具体任务 | 产出物 |
|------|------|----------|--------|
| 第 1 周 | **数据准备** | 确定 5~8 个核心数据源，编写爬虫脚本，提取并清洗第一批文档（至少 100 条切片） | 知识库 JSON/Markdown 文件包 |
| 第 2 周 | **RAG 核心跑通** | 搭建 ChromaDB，编写向量化脚本；实现 `retrieve_and_generate()` 函数，本地测试问答效果 | 可命令行交互的 RAG 原型 |
| 第 3 周 | **后端 API 开发** | 用 FastAPI 封装 `/chat` 接口，加入日志记录、缓存逻辑；编写管理后台 API 骨架 | 可 curl 调用的 API 服务 |
| 第 4 周 | **小程序前端开发** | 设计对话界面，实现 `wx.request` 调用后端，支持流式输出展示（或一次性展示） | 小程序可与后端联调 |
| 第 5 周 | **管理后台（Web）** | 开发简易后台（Vue 或纯 HTML+JS），实现知识库的增删查、问答日志查看 | 管理员可手动更新知识库 |
| 第 6 周 | **优化与反馈闭环** | 加入点赞/点踩按钮，实现高频问题缓存，优化 Prompt 模板 | 回答质量明显提升 |
| 第 7 周 | **部署与测试** | Docker 化整个项目，部署到你个人网站服务器，配置 Nginx 代理和 SSL | 小程序可正式扫码访问 |
| 第 8 周 | **文档撰写与演示** | 编写软件工程实验报告，录制演示视频，准备答辩 | 完整项目交付 |

## 八、快速启动

1. **收集第一批数据**：打开你学校官网，找到“学生服务”“办事指南”“常见问题”等栏目，保存 5 个典型页面的 URL 和文字内容。
2. **搭建 Python 环境**：安装 `chromadb`, `sentence-transformers`, `fastapi`, `openai`（用于调 DeepSeek 兼容接口）。
3. **跑通最小 RAG Demo**：写一个 Jupyter Notebook，用上面收集的 5 个页面做知识库，问一个问题看能否检索到。
