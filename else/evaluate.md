# 校园百事通计划审视报告

## 一、整体可行性

**结论：可行，但有几处关键风险需要正视。**

### ✅ 优势明确的部分

**技术路线成熟**。RAG + 向量数据库 + LLM API 的组合是目前业界最主流的知识库问答方案，开源生态完整，坑基本都有人踩过。对于一个本科软工实验项目而言，这个选型既有技术含量，又不会过于激进。

**数据合规性清晰**。只爬公开页面，不碰个人信息，这个边界划得很好，后期申请学校背书也不会在数据层面遇到障碍。

**部署路径简单**。有现成的个人服务器，Docker Compose 一键部署，省去了云服务器选型和备案的麻烦。

### ⚠️ 需要正视的风险

**RAG 的回答质量上限取决于数据质量，而不是模型能力。** 这是新手最容易踩的坑。校园官网的 HTML 质量普遍很差：大量用图片代替文字（比如课表、地图、通知截图）、JavaScript 动态渲染的页面 BeautifulSoup 直接抓不到内容、PDF 扫描件无法提取文字。第一周"数据准备"的工作量在计划中被严重低估了，实际难度可能是计划的 3 倍。

**微信小程序的网络域名限制**。小程序的 `wx.request` 只能请求已在微信公众平台备案的 HTTPS 域名，且必须是已备案的域名（国内服务器 ICP 备案）。你的个人网站是否已有 ICP 备案？这是上线的硬性前提，不是技术问题，是政策问题，需要提前确认。

**流式输出在小程序端实现有额外成本**。计划中提到"流式输出"，但微信小程序目前对 SSE（Server-Sent Events）的支持很有限，需要用 `wx.request` 的 chunked 响应或者 WebSocket 来实现，调试成本不低，建议初版先做一次性返回，后期再迭代流式。

## 二、技术框架审视

### 后端框架：FastAPI ✅ 合理

对于异步 IO 密集型场景（等待 LLM API 返回），FastAPI + asyncio 是正确选择。唯一补充：DeepSeek 的 Python SDK 直接支持 `async`，可以配合 `asyncio.gather` 实现并发请求，不需要额外引入 Celery。初期完全不需要多 Worker，单 Worker 足够。

### 向量数据库：ChromaDB ✅ 合理，但需注意一点

ChromaDB 嵌入式模式对于 5000 条以内的切片完全够用，选型合理。但需要注意：ChromaDB 在 Docker 中以持久化模式运行时，数据目录要挂载到宿主机，否则容器重启后数据丢失。这是个低级但致命的配置错误，新手经常犯。

### 嵌入模型：bge-small-zh ✅ 合理

`BAAI/bge-small-zh-v1.5` 是中文轻量嵌入模型的最佳选择之一，512 维、CPU 可跑、效果不差。但需要补充一点：**首次加载模型需要从 HuggingFace 下载约 400MB 的权重文件**，国内服务器访问 HuggingFace 可能需要镜像源（`hf-mirror.com`），部署时要提前处理，否则会卡在这一步很久。

### 大模型 API：DeepSeek ✅ 合理

DeepSeek 兼容 OpenAI 接口格式，直接用 `openai` SDK 改 `base_url` 就能调用，迁移成本极低。免费额度对于实验项目绰绰有余。唯一建议是在代码里做好 LLM 调用的超时和重试逻辑（比如超时 10s 后重试一次），因为 LLM API 偶尔会抖动。

### 数据库：SQLite → PostgreSQL ✅ 合理

初期用 SQLite 记录问答日志是完全正确的选择，后期有需要再迁移 PostgreSQL。用 SQLAlchemy ORM 写的话，换数据库只需改一行连接字符串。

### Redis 缓存 ⚠️ 可以推迟

Redis 对于校园级并发（峰值 QPS ≈ 1.2）完全不是瓶颈，文档里的计算结论是正确的。但 Redis 会增加部署复杂度（又多一个容器）。**建议初版用 Python 内存字典做简单缓存，第 6 周优化时再引入 Redis**，不要一开始就把架构搞复杂。

### 前端：微信小程序原生开发 ⚠️ 需要确认

原生小程序开发对于只做问答界面已经够用，但有几个前置成本容易被忽略：微信小程序需要注册（个人账号即可，但需要实名），开发工具、AppID、HTTPS 域名白名单配置等，首次上手大概需要半天到一天时间。如果对小程序开发不熟悉，这里的学习曲线要提前计入工期。

## 三、工期评估

**结论：整体偏乐观，第 1、2、4 周最危险。**

| 周次 | 计划评估 | 实际风险 |
|------|----------|----------|
| 第 1 周（数据准备） | 计划"确定数据源、编写爬虫、清洗 100 条切片" | **严重低估**。校园官网技术栈参差不齐，JS 渲染、图片代文字、PDF 扫描件这三个问题随便碰到一个都能消耗一整天。建议把数据准备扩展到 1.5~2 周，或者第一周只做最干净的数据源（比如纯文字 HTML 页面），先跑通 RAG，脏数据留后面处理。 |
| 第 2 周（RAG 核心） | 计划"搭建 ChromaDB、向量化、本地测试" | **基本合理**，但"测试问答效果"这一步容易陷入无休止调优。建议设定一个明确的完成标准：用 10 个预设问题测试，70% 以上能给出相关回答就进入下一阶段，不要追求完美。 |
| 第 3 周（后端 API） | FastAPI 封装接口 | **合理**，FastAPI 上手快，这周基本可以按计划完成。 |
| 第 4 周（小程序前端） | 对话界面 + 联调 | **偏乐观**。如果是第一次做小程序，WXML/WXSS 的布局模型和 Web 不完全一样，`wx.request` 的异步回调有一定学习成本。建议给联调多留 2~3 天，跨域/HTTPS/域名白名单问题几乎必定出现。 |
| 第 5 周（管理后台） | Web 后台 | **可以简化**。初版管理后台用纯 HTML+JS 直接调 FastAPI 接口就够了，不需要 Vue，省去框架学习成本。 |
| 第 6~8 周 | 优化、部署、文档 | **整体合理**，但文档撰写（软工实验报告）容易被低估，一份完整的软工报告从初稿到定稿通常需要 3~5 天。 |

**总体建议**：如果是 8 周硬性截止，把"选课评价模块"坚决推到二期，只交付核心问答功能，把省出来的时间用于数据质量和测试报告。

## 四、补充建议

### 关于后期学号登录功能

拿到学校背书后接入学号认证，技术上建议走 **OAuth / CAS 协议**（大多数高校的统一身份认证平台都支持 CAS），而不是直接让用户输入学号密码。原因是：直接处理学号密码会让你的系统承担用户凭证的安全责任，一旦出问题后果很严重；CAS 只需要对接一个重定向流程，用户在学校官方页面输入密码，你的系统只收到一个认证 token，安全责任完全在学校侧。

### 关于知识库的持续更新

计划中提到"每周手动或 Cron 执行爬虫"，这个策略初期完全可以，但需要额外考虑**增量更新**的问题：当某个页面更新后，对应的旧切片需要被替换，而不是累积。ChromaDB 支持按 `id` 更新文档，建议用页面 URL 的哈希值作为文档 ID，爬取时若 ID 已存在则覆盖，这样能避免知识库出现新旧矛盾的信息。

### 关于 Prompt 设计

这是影响用户体验最直接的环节，却在计划中着墨不多。几个实用建议：

一是在 Prompt 中明确要求模型**在回答末尾列出引用来源**，而不是混在正文里，这样小程序可以把来源单独渲染成可点击链接。二是加入一条兜底指令："若检索到的参考资料不足以回答问题，请明确告知用户'我暂时没有这方面的信息'，不要编造。"这条指令能有效降低幻觉风险。三是可以考虑在 Prompt 中加入当前日期，让模型在回答时序相关问题（比如"现在可以选课吗"）时能给出正确的时间判断。

### 关于选课评价模块的定位

这个功能模块的性质和百事通核心功能有本质差别：百事通是**信息检索**（查已有信息），选课评价是**UGC 内容平台**（用户生成内容）。后者需要用户身份系统（防刷）、内容审核、评分统计等，复杂度高出一个量级。建议明确把它定位为"二期功能，需要学校背书和登录系统作为前提"，不要在一期开发中为它预留任何接口，专注做好核心问答。

---

**一句话总结**：这个计划方向正确、技术选型合理，最大的风险在于数据准备被低估和小程序端的联调成本。建议第一周就去摸清楚 2~3 个目标官网的实际 HTML 结构，根据实际情况再决定要不要调整工期。有任何具体环节需要深入讨论（比如爬虫策略、Prompt 模板、ChromaDB 配置），可以直接提。
